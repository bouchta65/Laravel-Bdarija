Laravel Project Structure

A detailed guide to understanding each folder and file

Project Overview

Laravel follows the MVC (Model-View-Controller) pattern and organizes its files in a way that promotes clean, maintainable code. Let's explore each directory and understand its purpose in the Laravel ecosystem.

laravel-project/
├── app/
├── bootstrap/
├── config/
├── database/
├── public/
├── resources/
├── routes/
├── storage/
├── tests/
├── vendor/
├── .env
├── .env.example
├── artisan
├── composer.json
└── package.json

This is the typical structure of a Laravel project. We'll explain each directory and file in detail below, focusing on their specific roles in the application lifecycle.

1 app/ Directory

The app/ directory contains the core code of your application. This is where most of your application's logic lives.

app/Http/

Contains controllers, middleware, and form requests.

app/Http/
├── Controllers/   # Application controllers
├── Middleware/    # HTTP middleware filters
├── Requests/      # Form request validation classes
└── Kernel.php     # HTTP kernel, registers middleware

Controllers handle HTTP requests and return responses. Middleware provides a filtering mechanism for HTTP requests.

app/Models/

Contains Eloquent model classes that interact with your database.

app/Models/
├── User.php      # User model (included by default)
└── ...           # Your custom models

Models represent database tables and provide an elegant way to interact with your data. They're typically used for CRUD operations and defining relationships between data.

app/Providers/

Contains service providers that bootstrap your application.

app/Providers/
├── AppServiceProvider.php      # Main service provider
├── AuthServiceProvider.php     # Authentication services
├── BroadcastServiceProvider.php # Broadcasting configuration
├── EventServiceProvider.php    # Event listener mappings
└── RouteServiceProvider.php    # Route configuration

Service providers are the central place to configure your application. They register services in the container, bind interfaces to implementations, and perform other setup tasks.

Other app/ Directories

app/
├── Console/          # Custom Artisan commands
├── Exceptions/       # Exception handling
├── Jobs/             # Queueable jobs
├── Listeners/        # Event listeners
├── Mail/             # Mailable classes
├── Notifications/    # Notification classes
├── Policies/         # Authorization policies
├── Rules/            # Custom validation rules
└── Services/         # Service classes (custom)

These directories organize different aspects of your application's functionality, from background jobs to notifications and more.

2 bootstrap/ Directory

The bootstrap/ directory contains files that bootstrap the framework. This includes the app.php file which creates the application instance.

bootstrap/
├── app.php         # Creates the Laravel application
├── cache/          # Framework generated cache files
└── ...

Note: You'll rarely need to modify anything in this directory. The bootstrap/cache directory should be writable by your web server.

3 config/ Directory

The config/ directory contains all of your application's configuration files.

config/
├── app.php         # Application configuration
├── auth.php        # Authentication configuration
├── broadcasting.php # Broadcasting configuration
├── cache.php       # Cache configuration
├── database.php    # Database configuration
├── filesystems.php # File storage configuration
├── mail.php        # Email configuration
├── queue.php       # Queue configuration
├── services.php    # Third-party services configuration
├── session.php     # Session configuration
└── view.php        # View configuration

Each file returns an array of configuration options that Laravel uses throughout the application. The values typically come from environment variables (.env file) for security and flexibility across different environments.

Tip: You can access configuration values using the config() helper function: config('app.name')

4 database/ Directory

The database/ directory contains your database migrations, factories, and seeders.

database/
├── factories/      # Model factories for testing/seeding
├── migrations/     # Database migrations
├── seeders/        # Database seeders
└── .gitignore

migrations/

Migrations are like version control for your database. They allow you to define and share your database schema.

// Example migration file
public function up()
{
    Schema::create('users', function (Blueprint $table) {
        $table->id();
        $table->string('name');
        $table->string('email')->unique();
        $table->timestamp('email_verified_at')->nullable();
        $table->string('password');
        $table->rememberToken();
        $table->timestamps();
    });
}

Run migrations with php artisan migrate

seeders/

Seeders fill your database with test data. They're useful for development and testing.

// Example seeder file
public function run()
{
    \App\Models\User::factory(10)->create();
}

Run seeders with php artisan db:seed

factories/

Factories generate fake model instances for testing and seeding.

// Example factory definition
public function definition()
{
    return [
        'name' => $this->faker->name(),
        'email' => $this->faker->unique()->safeEmail(),
        'password' => '$2y$10$92IXUNpkjO0rOQ5byMi.Ye4oKoEa3Ro9llC',
    ];
}

Use factories in tests or seeders: User::factory()->count(5)->create()

5 public/ Directory

The public/ directory contains the entry point to your application (index.php) and all publicly accessible assets.

public/
├── index.php       # Application entry point
├── favicon.ico     # Site favicon
├── robots.txt      # Robots control file
├── .htaccess       # Apache configuration
├── css/            # Compiled CSS
├── js/             # Compiled JavaScript
└── images/         # Public images

Important: This is the only directory that should be publicly accessible in your web server configuration. All other Laravel code should be outside the public web root.

6 resources/ Directory

The resources/ directory contains uncompiled assets like SASS, JavaScript, and views.

resources/
├── css/            # CSS or SASS files
├── js/             # JavaScript files
├── lang/           # Language/localization files
└── views/          # Blade template files

views/

Contains Blade template files that display your application's HTML.

resources/views/
├── layouts/        # Layout templates
├── components/     # Reusable view components
├── partials/       # Partial views
└── pages/          # Page-specific views

Blade templates mix HTML with PHP directives for dynamic content.

js/ and css/

Source files for JavaScript and CSS that will be compiled using Laravel Mix (webpack).

// Example webpack.mix.js configuration
mix.js('resources/js/app.js', 'public/js')
   .sass('resources/sass/app.scss', 'public/css');

Compile these assets using: npm run dev or npm run production

7 routes/ Directory

The routes/ directory contains all the route definitions for your application.

routes/
├── web.php         # Web routes (HTML, cookies)
├── api.php         # API routes (JSON, stateless)
├── console.php     # Console commands
├── channels.php    # Websocket channels
└── ...

web.php

Contains routes for your web interface. These routes are stateful and have access to session and cookies.

// Example web routes
Route::get('/', function () {
    return view('welcome');
});

Route::get('/dashboard', [DashboardController::class, 'index'])
    ->middleware(['auth'])
    ->name('dashboard');

api.php

Contains routes for your API. These routes are stateless and typically return JSON.

// Example API routes
Route::middleware('auth:sanctum')->group(function () {
    Route::get('/user', function (Request $request) {
        return $request->user();
    });
    
    Route::apiResource('posts', PostController::class);
});

API routes are prefixed with '/api' and use the 'api' middleware group by default.

8 storage/ Directory

The storage/ directory contains application-generated files like logs, cache files, and uploaded files.

storage/
├── app/            # Application-generated files
│   └── public/     # Public file uploads
├── framework/      # Framework-generated files
│   ├── cache/      # Cache files
│   ├── sessions/   # Session files
│   └── views/      # Compiled Blade templates
└── logs/           # Application logs

Note: When storing user-uploaded files, you'll typically use the storage/app/public directory. To make these files accessible from the web, create a symbolic link from public/storage to storage/app/public using php artisan storage:link.

9 tests/ Directory

The tests/ directory contains automated tests for your application.

tests/
├── Feature/        # Feature tests
├── Unit/           # Unit tests
├── TestCase.php    # Base test case class
└── CreatesApplication.php # Application factory trait

Feature/

Contains feature tests that test larger portions of your code, often making HTTP requests to your application.

// Example feature test
public function test_user_can_view_posts()
{
    $user = User::factory()->create();
    $post = Post::factory()->create();
    
    $response = $this->actingAs($user)
                      ->get('/posts');
    
    $response->assertOk()
             ->assertSee($post->title);
}

Unit/

Contains unit tests that test individual components in isolation.

// Example unit test
public function test_post_can_be_created()
{
    $post = Post::factory()->create();
    
    $this->assertDatabaseHas('posts', [
        'id' => $post->id,
        'title' => $post->title
    ]);
}

Run tests with php artisan test

10 vendor/ Directory

The vendor/ directory contains Composer dependencies, including the Laravel framework itself.

vendor/
├── bin/            # Executable binaries
├── composer/       # Composer files
├── laravel/        # Laravel framework
└── ... (other packages)

Important: Never modify files in the vendor directory directly. This directory is managed by Composer and any changes will be overwritten when dependencies are updated.

11 Root Files

.env and .env.example

Environment configuration files. The .env file contains your application's environment-specific settings like database credentials, API keys, etc.

# Example .env file
APP_NAME=Laravel
APP_ENV=local
APP_KEY=base64:abcdefghijklmnopqrstuvwxyz123456=
APP_DEBUG=true
APP_URL=http://localhost

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=laravel
DB_USERNAME=root
DB_PASSWORD=

Security Note: The .env file contains sensitive information and should never be committed to version control. The .env.example file is a template that you can commit safely.

artisan

The command-line interface for Laravel. It provides many helpful commands for development.

# Common Artisan commands
php artisan list                  # List all commands
php artisan make:controller       # Create a controller
php artisan make:model            # Create a model
php artisan make:migration        # Create a migration
php artisan migrate               # Run database migrations
php artisan serve                 # Start development server

Install dependencies with composer install and npm install

Putting It All Together

Laravel's directory structure follows a logical organization that separates concerns and makes it easy to maintain your application:

Core Application Logic

  • app/ - Models, controllers, middleware
  • routes/ - URL definitions
  • config/ - Configuration files

Frontend & Views

  • resources/ - Views, JS, CSS, language files
  • public/ - Web-accessible compiled assets

Data & Storage

  • database/ - Migrations, factories, seeders
  • storage/ - Logs, cache, uploads

Tests & Dependencies

  • tests/ - Unit & feature tests
  • vendor/ - Composer packages
  • node_modules/ - npm packages

Next Steps

Now that you understand Laravel's directory structure, you can start building your application with confidence. Remember these key workflows:

  • Define routes in routes/web.php or routes/api.php
  • Create controllers in app/Http/Controllers/
  • Define models in app/Models/
  • Create views in resources/views/
  • Run database migrations with php artisan migrate
